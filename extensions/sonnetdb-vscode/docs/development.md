# VS Code Extension Development

This repository-only document describes the internal structure, design boundaries, verification, and packaging workflow for the SonnetDB VS Code extension. End-user instructions belong in the extension `README.md`, which is rendered on Visual Studio Marketplace.

## Prerequisites

- Node.js and npm compatible with the checked-in lockfile
- .NET 10 SDK
- Visual Studio Code 1.100 or later for Extension Host smoke tests

Install dependencies from `extensions/sonnetdb-vscode`:

```powershell
npm ci
```

## Working principles

- Keep the extension remote-first and reuse SonnetDB Server contracts instead of embedding the database engine in the Node extension host.
- Store connection metadata in VS Code global state and bearer tokens in `SecretStorage`.
- Start local database access through an extension-managed SonnetDB Server and reuse the same HTTP client as remote mode.
- Keep multi-model Explorer and preview features read-only. Full editing, import/export, approval, and governance remain Web Admin / Studio responsibilities.
- Reuse the web management implementation as the reference for NDJSON parsing, SQL dialect keywords, schema completion, chart and trajectory rendering, and Copilot request payloads.
- Keep the C# parser sidecar stateless. It receives SQL validation requests over JSON-RPC 2.0 with standard LSP `Content-Length` framing and must not receive credentials or database paths.
- Preserve lightweight TypeScript diagnostics whenever the sidecar or .NET runtime is unavailable.

See [architecture.md](./architecture.md) for runtime boundaries and [api-contract.md](./api-contract.md) for the Server endpoints consumed by the extension.

## Brand assets

The canonical SonnetDB logo is stored at `web/public/favicon.svg` and `web/public/favicon.png` in the repository root. The files under `media/` are package-local synchronized copies required by VS Code and Marketplace; do not redraw an extension-specific logo.

Marketplace requires a package-local PNG. Keep `media/sonnetdb.png` at 256 x 256 and make sure `.vscodeignore` does not exclude it.

## Verification

Run unit tests and TypeScript compilation:

```powershell
npm test
```

Run the C# Language Server tests from the extension directory:

```powershell
dotnet test ../../tests/SonnetDB.LanguageServer.Tests/SonnetDB.LanguageServer.Tests.csproj
```

Run the isolated VS Code Extension Host smoke test:

```powershell
npm run test:host
```

The smoke runner creates isolated user-data and extension directories, starts a real VS Code Extension Host, verifies activation and editor features, and removes the temporary profile after completion.

## Package verification

Build the portable Language Server and VSIX:

```powershell
npm run package:vsix
npx vsce ls --tree
Get-FileHash dist/sonnetdb-vscode.vsix -Algorithm SHA256
```

Confirm the VSIX contains:

- `out/`
- `snippets/`
- `media/sonnetdb.png`
- `media/sonnetdb.svg`
- portable `language-server/*.dll` files

Confirm the VSIX does not contain:

- PDB files or TypeScript source
- `node_modules/`
- `docs/` or contributor verification instructions
- credentials, `.env` files, or browser profiles

## Release rules

- Every Marketplace update uses a new immutable semantic version in `package.json` and `package-lock.json`.
- Update `CHANGELOG.md`, rebuild the VSIX, and repeat the complete verification flow for every release correction.
- Update the existing `iotsharp.sonnetdb-vscode` Marketplace item. Do not create another extension with the same name.
- Compare the local SHA256 with Marketplace metadata after processing completes.
- Never store a Marketplace PAT or browser profile in the repository.
